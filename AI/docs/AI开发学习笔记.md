## 先建立整体认知

如果只记一件事，可以记下面这条链路：

```text
AI = 总体概念
LLM = AI 中最常用来处理语言、代码和知识工作的核心模型
Agent = 带有目标、流程、工具调用能力的 AI 执行体
SKILLS = 给 Agent 的“工作方法包”
MCP = 让 Agent 接外部工具和数据的标准协议
```

再压缩一点：

- `AI` 决定“机器能不能做智能任务”
- `LLM` 决定“机器能不能理解和生成语言/代码”
- `SKILLS` 决定“机器做事有没有方法”
- `MCP` 决定“机器能不能连到真实世界里的工具和数据”

所以，所谓“用 AI 开发”，并不是单纯让模型吐代码，而是让模型在合适的流程、上下文和工具连接下参与软件工程。

## 什么是 AI

### 1. 定义

`AI`，即 Artificial Intelligence，人工智能，是指让计算机执行原本通常需要人类智能才能完成的任务，例如：

- 理解自然语言
- 识别图像和语音
- 做分类、预测和推荐
- 生成文本、代码、图片、音频
- 根据目标进行推理、规划和执行

### 2. AI 不是单一技术

AI 是一个大类，下面包含很多分支：

- `机器学习（Machine Learning）`：通过数据学习规律，而不是完全写死规则
- `深度学习（Deep Learning）`：用神经网络处理更复杂的问题
- `生成式 AI（Generative AI）`：生成文本、代码、图像、音频等内容
- `大语言模型（LLM）`：生成式 AI 中最常用于文本和代码的一类模型

它们之间的关系可以这样理解：

```text
AI
└── 机器学习
    └── 深度学习
        └── 生成式 AI
            └── 大语言模型（LLM）
```

### 3. AI 的核心价值

站在开发者视角，AI 的价值通常不在“完全替代人”，而在下面几件事：

- 降低信息处理成本：快速读文档、总结需求、解释代码
- 降低生成成本：快速写样板代码、测试、脚本、文档
- 提高沟通效率：把模糊问题转成更清晰的方案或任务
- 提高试错速度：先生成一个可运行版本，再迭代修正

### 4. AI 的边界

AI 很强，但不等于总是正确。常见边界包括：

- 可能“胡说八道”，即幻觉（hallucination）
- 可能生成能运行但设计很差的代码
- 可能缺少你项目的真实上下文
- 可能忽略隐藏约束，例如兼容性、安全、性能、合规
- 可能说得很像对的，但其实没验证

所以，AI 适合作为“高产出的助手”，不适合作为“不需要校验的最终裁决者”。

## 什么是大语言模型

### 1. 定义

`大语言模型（LLM, Large Language Model）` 是一种在海量文本和代码上训练出来的模型。它最核心的能力，是根据上下文预测后续最合理的内容。

从结果上看，它能做很多事情：

- 问答
- 总结
- 翻译
- 写作
- 代码生成
- 代码解释
- 错误排查
- 结构化输出

### 2. 一个足够实用的理解方式

不要把 LLM 想成数据库，更不要把它想成一个“永远知道正确答案的专家”。

更准确的理解是：

- 它是一个非常强的“语言和模式建模器”
- 它能根据上下文做近似推理
- 它很擅长把已有知识重组为有用输出
- 它在没有外部验证时，可能会把“不确定”包装成“确定”

### 3. LLM 是怎么工作的

可以用一个简化版本来理解：

1. `预训练（Pretraining）`
   模型先在海量文本和代码上学习语言模式、常识、结构、风格和部分知识。
2. `指令对齐（Instruction Tuning / Alignment）`
   模型进一步学习如何按人类要求回答问题。
3. `推理阶段（Inference）`
   用户输入一个问题，模型结合上下文，逐步生成 token，最终形成回答。

### 4. LLM 的几个关键概念

#### Token

- 模型处理的最小文本单位，不一定等于一个汉字或一个单词
- 上下文越长，消耗的 token 越多
- 你给模型的指令、代码、日志、对话历史都会占用 token

#### Context Window

- 模型一次能“看到”的上下文范围
- 超过窗口大小的内容，模型通常无法同时参考
- 所以长项目里，信息筛选和上下文管理非常重要

#### Prompt

- 你给模型的输入指令
- Prompt 不只是“提问题”，更重要的是定义任务、上下文、限制和输出格式

#### Temperature

- 控制输出的随机性
- 较低时更稳定，更适合代码、总结、结构化输出
- 较高时更发散，更适合创意生成

#### Tool Use

- 模型不只是“说”，还可以“调用工具”
- 例如读文件、查数据库、发请求、执行命令、调用 API
- 一旦模型可以用工具，它就更接近一个 `Agent`

#### RAG

- `RAG` 是 `Retrieval-Augmented Generation`，中文常译为“检索增强生成”
- 它的思路是：先从外部知识库里检索相关内容，再把检索结果作为上下文交给模型生成答案
- 它常用于解决模型知识过期、缺少私有知识、需要引用项目文档等问题

#### Fine-tuning

- `Fine-tuning` 即微调，是在已有模型基础上继续训练，让模型更符合某类任务、风格或输出模式
- 微调更适合改变模型的“行为倾向”或“领域适配”
- 它不适合替代实时知识接入，也不适合替代日常工具调用

### 5. LLM 擅长什么

- 处理自然语言和代码
- 快速归纳信息
- 生成第一版方案、代码、文档
- 把非结构化输入变成结构化结果
- 在你提供了足够上下文后，参与复杂的工程任务

### 6. LLM 不擅长什么

- 在缺少上下文时准确理解你项目的特殊约束
- 在未经验证时保证事实正确
- 自动理解隐藏需求
- 代替真实测试、真实部署和真实线上观察

### 7. 开发者最该知道的一个事实

LLM 的价值并不主要来自“它会不会背答案”，而是来自：

- 它能否拿到正确上下文
- 它能否按正确流程工作
- 它能否连接真实工具
- 你是否能验证它的输出

这就引出了 `SKILLS` 和 `MCP`。

### 8. Prompt、RAG、微调分别适合解决什么问题

这是开发者经常混淆的三个层次：

- `Prompt`：解决“这次任务怎么描述”
- `RAG`：解决“模型这次回答时需要哪些外部资料”
- `微调`：解决“模型整体风格或行为应该如何更稳定地表现”

一个简单判断方式：

- 信息是临时的、任务级的，用 `Prompt`
- 信息是外部的、可检索的、可能变化的，用 `RAG`
- 行为模式是长期稳定需求，用 `微调`

## 什么是 SKILLS

### 1. 这里的 SKILLS 不是泛指“技能”

在 Agent/Codex/Claude Code 这类环境里，`SKILLS` 通常指一套可复用、模块化、面向任务的能力包或工作流说明。

你可以把它理解成：

> 给 AI 的“标准作业流程（SOP）+ 任务说明书 + 补充资源”

它不是模型参数的一部分，而是运行时加载的任务方法。

### 2. SKILLS 解决什么问题

如果没有 Skill，模型常常会：

- 直接开始做事，但没想清楚流程
- 跳过关键检查步骤
- 输出看起来完整，但工程上不严谨
- 在复杂任务里缺少稳定做法

Skill 的作用，就是把“某类任务该怎么做”写清楚，让 Agent 少走弯路。

### 3. 一个 Skill 通常包含什么

在当前这类 Codex 环境里，Skill 常见结构是：

```text
skill-name/
├── SKILL.md          # 核心说明，定义何时触发、怎么做
├── scripts/          # 可执行脚本
├── references/       # 参考文档
└── assets/           # 模板、静态资源等
```

其中最核心的是 `SKILL.md`。

### 4. Skill 的作用方式

可以把 Skill 理解成“给模型临时补一套方法论”：

- 普通 Prompt：告诉模型“这次要做什么”
- Skill：告诉模型“这类事情通常应该怎么做”

例如：

- `brainstorming`：先澄清需求，再收敛方案
- `test-driven-development`：先写测试，再实现
- `systematic-debugging`：先定位问题，再提出修复
- `requesting-code-review`：完成后做审查

### 5. Skill 的价值

Skill 的真正价值不在“多一个说明文件”，而在于它把一些容易被忽略的工程纪律固化下来了，例如：

- 先理解需求再动手
- 先测试再实现
- 先验证再声称修复完成
- 先做 review 再结束任务

### 6. Skill 和 Prompt 的区别

| 项目 | Prompt | Skill |
| --- | --- | --- |
| 目的 | 说明当前任务 | 规范一类任务的做法 |
| 生命周期 | 当前对话或当前请求 | 可长期复用 |
| 内容 | 任务目标、上下文、输出要求 | 触发条件、步骤、约束、工具使用方式 |
| 稳定性 | 较临时 | 较稳定 |

### 7. Skill 和脚本的区别

- `脚本` 更偏执行逻辑，强调确定性
- `Skill` 更偏工作方法，强调流程和判断
- 两者经常结合使用：Skill 负责指导，脚本负责稳定执行

### 8. 怎样判断什么时候需要 Skill

当任务符合下面任意一种情况时，Skill 很有价值：

- 这个任务会反复做
- 这个任务容易漏步骤
- 这个任务需要固定流程
- 这个任务涉及公司内部知识、规则或工具
- 你希望不同 Agent 在相似任务上表现更稳定

## 什么是 MCP

### 1. 定义

`MCP`，即 `Model Context Protocol`，可以理解为：

> 一种让模型以统一方式连接外部工具、数据源和上下文资源的标准协议

如果说 LLM 是大脑，MCP 更像是让这个大脑接上“手和眼”的接口层。

### 2. MCP 解决什么问题

如果没有统一协议，不同工具的接法会非常分散：

- GitHub 一套接口
- 数据库一套接口
- 文件系统一套接口
- 内部平台又一套接口

这样会导致模型很难稳定地使用工具。

MCP 的目标是把这些接入方式标准化，让模型可以更一致地发现、读取和调用外部能力。

### 3. 在 MCP 视角下，常见对象有哪些

#### MCP Server

- 对外暴露能力的服务端
- 可以理解为“某个外部系统对 AI 的标准化接入口”

例如，一个 GitHub MCP Server 可能提供：

- 读仓库
- 查 issue
- 创建 PR
- 读文件

#### Tools

- 可供模型调用的动作
- 例如：创建 issue、查询 schema、搜索文档、读日志

#### Resources

- 可供模型读取的静态或半静态上下文资源
- 例如：数据库 schema、项目文档、内部规范

#### Templates

- 参数化资源模板
- 用于按需获取某一类上下文

### 4. MCP 的价值

MCP 最大的价值是把“模型知道什么”扩展为“模型能接触到什么”。

举例：

- 没有 MCP 时，模型只能根据已有上下文回答
- 有了 MCP 后，模型可以去读真实文档、真实数据、真实代码、真实工单

这会显著提升：

- 准确性
- 可操作性
- 自动化程度

### 5. 一个开发场景下的例子

假设你问 AI：

> 为什么这个接口在生产环境报错？

如果只有模型本身，它最多给你一些推测。

如果模型通过 MCP 能访问：

- 日志系统
- 监控系统
- 仓库代码
- 线上配置

那它就可以：

1. 查询错误日志
2. 读取相关代码
3. 对比配置差异
4. 给出更可信的排查结论

### 6. MCP 和普通 API 集成的区别

你可以把 MCP 看成“面向模型的统一协议层”，而不是某一个具体 API。

- `普通 API 集成`：开发者自己处理调用细节
- `MCP`：把工具能力用标准方式暴露给模型

换句话说，MCP 更像“AI 工具生态的通用插座”。

### 7. Skill 和 MCP 的区别

- `Skill` 解决“怎么做”
- `MCP` 解决“能连什么、能用什么”

两者配合时效果最好：

- Skill 规定流程
- MCP 提供真实工具与数据
- 模型负责理解、推理和生成

### 8. MCP 和 RAG 的区别

这两个词也很容易混在一起，但它们并不是一回事。

| 项目 | MCP | RAG |
| --- | --- | --- |
| 本质 | 协议/接入标准 | 一种检索增强方案 |
| 目标 | 让模型连接工具与资源 | 让模型回答前先拿到相关资料 |
| 范围 | 可以连工具、资源、模板、动作 | 主要围绕检索知识并注入上下文 |
| 是否能执行动作 | 可以 | 一般不负责执行动作 |

可以把它理解成：

- `RAG` 更像“先查资料再回答”
- `MCP` 更像“给模型一套标准化的工具和资源接口”

两者可以结合：

- 用 `MCP` 暴露知识库查询工具
- 用 `RAG` 组织检索结果并增强回答

## 这些概念之间是什么关系

可以用下面这张表一起看：

| 概念 | 核心问题 | 作用 |
| --- | --- | --- |
| AI | 机器能否做智能任务 | 总体概念 |
| LLM | 机器能否理解和生成语言/代码 | 核心模型 |
| Agent | 机器能否围绕目标持续执行任务 | 执行形态 |
| Prompt | 这次到底要做什么 | 当前任务指令 |
| Skill | 这类任务应该怎么做 | 可复用工作流 |
| MCP | 机器可以连接什么工具和数据 | 外部能力接口 |

也可以记成一条执行链路：

```text
用户目标
  -> Prompt 说明当前任务
  -> LLM 负责理解和推理
  -> Skill 约束任务流程
  -> MCP 提供工具和数据
  -> Agent 执行、反馈、迭代
```

## 如何使用 AI 进行开发

### 1. 不要只把 AI 当“代码生成器”

AI 在开发中的价值远不止写代码。它可以参与整个软件工程流程：

- 需求理解
- 方案设计
- 代码实现
- 测试编写
- Bug 排查
- Code Review
- 文档整理
- 自动化脚本编写

### 2. AI 最适合做什么样的开发工作

#### 需求澄清

你可以让 AI 帮你：

- 把模糊需求拆成明确任务
- 识别缺失信息
- 输出验收标准
- 比较不同方案

#### 架构和设计讨论

你可以让 AI：

- 比较 2 到 3 种实现方案
- 分析模块边界
- 识别潜在风险
- 提出最小可行实现路径

#### 编码

你可以让 AI：

- 生成样板代码
- 根据已有代码风格补全实现
- 写脚本和工具函数
- 补测试
- 做小范围重构

#### 调试

你可以把下面信息交给 AI：

- 报错信息
- 日志
- 相关代码片段
- 复现步骤

然后让它：

- 归纳可能原因
- 提出验证步骤
- 给出修复方案

#### Review 和文档

你可以让 AI：

- 审查代码风险
- 检查漏测场景
- 生成 README、设计说明、接口文档、变更说明

### 3. 一个高效的 AI 开发流程

下面是一套非常实用的协作流程：

1. `先定义目标`
   说明你要实现什么，不要只说“帮我写一下”
2. `补上下文`
   提供相关代码、文件路径、接口定义、错误信息、限制条件
3. `先要方案，再写代码`
   对复杂任务，先让 AI 给出实现思路和风险点
4. `小步执行`
   不要一次让 AI 改十几个模块，先做一个可验证的小改动
5. `强制验证`
   要求 AI 运行测试、说明验证范围、列出未验证风险
6. `人工复核`
   重点看边界条件、安全、兼容性、异常处理和是否真的满足需求

### 4. 给 AI 提需求时，建议至少包含 5 个元素

#### 目标

你到底要什么结果。

#### 上下文

涉及哪些文件、模块、接口、日志、约束。

#### 限制

哪些东西不能改，必须兼容什么，是否允许引入新依赖。

#### 验收标准

什么叫完成，什么叫正确。

#### 输出方式

你希望它先给方案、直接改代码、还是先写测试。

### 5. 一个通用提问模板

```text
目标：
我要实现/修复什么？

上下文：
相关文件、模块、接口、日志是什么？

限制：
不能改什么？必须兼容什么？能否新增依赖？

验收标准：
什么结果算完成？如何验证？

执行要求：
先给计划，再实现；或直接改代码并运行测试。
```

### 6. 开发中几个非常有用的 AI 使用姿势

#### 姿势一：让 AI 先拆任务

适合：

- 需求比较大
- 自己一时不知道从哪下手
- 任务涉及多个文件或多个模块

示例：

```text
请把这个需求拆成可执行的小任务，并标出依赖关系、风险点和建议实施顺序。
```

#### 姿势二：让 AI 先写测试

适合：

- 修 bug
- 补边界条件
- 防回归

示例：

```text
请先根据这个 bug 描述写一个失败测试，不要先改实现。确认测试能稳定复现后，再给出最小修复。
```

#### 姿势三：让 AI 解释现有代码

适合：

- 接手陌生项目
- 阅读老代码
- 理解复杂函数或流程

示例：

```text
请解释这个模块的职责、调用链、输入输出和潜在风险，并指出最值得重构的地方。
```

#### 姿势四：让 AI 做 review，而不是只让它“看一眼”

适合：

- 合并前检查
- 排查隐藏 bug
- 识别测试缺口

示例：

```text
请按代码审查的方式检查这次改动，优先指出 bug、行为回归风险、异常路径和缺失测试。
```

### 7. AI 开发的三个原则

#### 原则一：上下文比“提示词技巧”更重要

很多时候，AI 回答差不是因为模型弱，而是因为你没有给它关键上下文。

#### 原则二：验证比生成更重要

真正贵的不是“写出第一版代码”，而是“确认第一版代码真的对”。

#### 原则三：复杂任务必须拆解

任务越大，越不能一句话全交给 AI。要先拆，再做，再验证。

## 常见使用模式

### 1. 新功能开发

推荐流程：

1. 先让 AI 澄清需求和边界
2. 让 AI 提供 2 到 3 个实现方案
3. 选定方案后，让 AI 按模块拆任务
4. 先写测试，再逐步实现
5. 每一步都验证

### 2. Bug 修复

推荐流程：

1. 提供报错、日志、复现步骤
2. 让 AI 先分析原因并列出验证方法
3. 先写失败测试
4. 做最小修复
5. 跑测试并检查是否引入回归

### 3. 代码重构

推荐流程：

1. 先让 AI 说明当前代码的问题
2. 定义这次重构“不改变行为”的边界
3. 先补保护性测试
4. 分阶段重构，而不是一次性大改

### 4. 文档与知识沉淀

推荐流程：

1. 提供现有代码和背景
2. 让 AI 归纳模块职责、依赖和流程
3. 输出 README、架构说明或 onboarding 文档
4. 人工检查术语和事实准确性

## 常见误区

### 1. 任务描述过于模糊

例如：

- “帮我优化一下”
- “帮我写个功能”
- “这个不太对，你改改”

这类输入通常会导致输出发散。

更好的方式是明确：

- 要达成什么结果
- 涉及哪些范围
- 什么不能改
- 如何验证

### 2. 把 AI 当成无需验证的专家

AI 可以帮你节省很多时间，但它不是自动正确。

对代码输出至少要检查：

- 逻辑是否满足需求
- 测试是否充分
- 错误处理是否完整
- 是否引入兼容性或安全风险

### 3. 一次丢给 AI 太大的任务

大任务不拆分，常见后果是：

- 忽略细节
- 修改范围失控
- 输出看似完整，实际难以落地

### 4. 只关注“生成”，不关注“验证”

真正的开发效率，来自：

- 更快写出第一版
- 更快验证正确性
- 更快定位错误

如果只有第一步，效率提升并不稳定。

### 5. 不理解 Skill 和 MCP 的意义

只让模型“聊天式写代码”，很容易卡在：

- 缺流程
- 缺工具
- 缺上下文

而 Skill 和 MCP 正是在补这三件事。

## 建议的学习路径

### 第一阶段：先把概念分清

先理解下面几个词分别是什么意思：

- AI
- 机器学习
- 深度学习
- 生成式 AI
- LLM
- Agent
- Prompt
- Skill
- MCP

目标不是背定义，而是知道它们各自负责什么问题。

### 第二阶段：学会高质量提问

重点练习：

- 明确目标
- 补齐上下文
- 写清限制
- 定义验收标准
- 要求输出结构化结果

### 第三阶段：把 AI 用到日常开发里

从下面场景开始最容易见效：

- 解释代码
- 写测试
- 修小 bug
- 写脚本
- 做 review
- 整理文档

### 第四阶段：学习 Agent 化工作流

这一阶段开始关注：

- 如何让 AI 调工具
- 如何约束流程
- 如何分步骤执行
- 如何做验证闭环

这时你就会越来越理解 Skill 和 MCP 的价值。

### 第五阶段：沉淀自己的 Skill 和工具接入

当你发现某些任务反复出现时，就可以开始沉淀：

- 团队自己的 Skill
- 项目的标准 Prompt
- 通过 MCP 暴露给 AI 的内部文档、服务和工具

这会让 AI 从“偶尔帮忙”进化为“稳定生产力工具”。

## 术语速查

### AI

人工智能，能执行智能任务的总体概念。

### LLM

大语言模型，擅长处理文本和代码。

### Agent

以目标为导向，能够结合上下文、工具和流程持续执行任务的 AI 形态。

### Prompt

给模型的任务输入。

### Token

模型处理文本的基本单位。

### Context Window

模型一次可参考的上下文范围。

### Hallucination

模型生成了看起来像真的、但实际不正确的内容。

### Skill

给 Agent 的可复用任务工作流与方法说明。

### MCP

让模型标准化连接外部工具和上下文资源的协议。

## 最后总结

如果要用一句话总结这份文档，可以这样记：

> 用 AI 开发，不是把任务丢给一个会说话的模型，而是让模型在正确的上下文、流程和工具连接下，参与真实的软件工程。

再拆成四句话：

- `AI` 是总称
- `LLM` 是处理语言和代码的核心能力
- `SKILLS` 让 AI 做事更有方法
- `MCP` 让 AI 接触真实工具和真实数据

当你把这四层都理解清楚后，就不再只是“会用聊天机器人”，而是在真正学习如何构建和使用 AI 开发能力。
